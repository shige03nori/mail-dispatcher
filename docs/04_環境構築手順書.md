# 環境構築手順書

**システム名**: mail-dispatcher（メール配信管理システム）
**バージョン**: 1.1
**作成日**: 2026-03-02
**更新日**: 2026-03-14

---

> **この手順書の目的**
> mail-dispatcher を自分のパソコンで動かすための手順をまとめた文書です。
> IT 初心者の方でも迷わないよう、各ステップに「なぜこの手順が必要か」を記載しています。

---

## 環境構築の全体の流れ

1. 前提ソフトウェアの確認（Node.js / npm / Git）
2. リポジトリの取得（`git clone`）
3. パッケージインストール（`npm install`）
4. 環境変数の設定（`.env` ファイル作成）
5. データベースのセットアップ（`migrate` + `generate`）
6. 初期データの投入（`npm run seed`）
7. 開発サーバー起動（`npm run dev`）
8. ブラウザで `http://localhost:3000` にアクセスして確認

## 1. 前提条件

### 1.1 必要なソフトウェア

| ツール | 必要バージョン | 確認コマンド | 使用用途 |
|--------|-------------|-------------|---------|
| Node.js | 20.x 以上 | `node -v` | JavaScript の実行環境 |
| npm | 10.x 以上 | `npm -v` | パッケージ（ライブラリ）管理 |
| Git | 任意 | `git --version` | ソースコードの管理・取得 |

> **「Node.js」とは？**
> JavaScript をサーバー側でも実行できるようにするソフトウェアです。
> このアプリの Web サーバーも Node.js 上で動きます。

> **「npm」とは？**
> Node.js のパッケージ（ライブラリ）を管理するツールです。
> `npm install` で必要なライブラリをまとめてダウンロードできます。

### 1.2 バージョン確認手順

コマンドプロンプト（またはターミナル）を開いて確認します。

```bash
node -v
# 出力例: v20.11.0  ← v20 以上であれば OK

npm -v
# 出力例: 10.2.4  ← 10 以上であれば OK

git --version
# 出力例: git version 2.43.0  ← バージョンが表示されれば OK
```

### 1.3 Windows 環境での注意事項

PowerShell でスクリプト実行エラーが出る場合は以下を試してください。

```powershell
# 現在の設定を確認
Get-ExecutionPolicy -List

# 変更（管理者権限不要）
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
```

> **Git Bash の使用を推奨**
> Windows では PowerShell ではなく **Git Bash** を使うとトラブルが少なくなります。
> Git for Windows をインストールすると Git Bash が同梱されています。

## 2. セットアップ手順

### 2.1 リポジトリを取得する

> **「リポジトリ」とは？**
> プロジェクトのファイル一式をまとめたものです。
> `git clone` でサーバーから自分のパソコンにコピーします。

```bash
# 任意のフォルダに移動（例: デスクトップ）
cd ~/Desktop

# リポジトリをクローン（URL は実際のものに変更してください）
git clone <リポジトリURL>

# クローンしたフォルダに移動
cd mail-dispatcher
```

実行後、`mail-dispatcher` という名前のフォルダが作成されます。

---

### 2.2 依存パッケージをインストールする

> **「依存パッケージ」とは？**
> このアプリが動くために必要な外部ライブラリのことです。
> `package.json` に一覧が記載されており、`npm install` で一括ダウンロードします。

```bash
npm install
```

実行後の出力例:
```
added 890 packages, and audited 891 packages in 45s
found 0 vulnerabilities
```

> **node_modules フォルダについて**
> インストールしたライブラリは `node_modules` フォルダに保存されます。
> このフォルダは Git で管理されないため、クローン後は必ず `npm install` を実行してください。

---

### 2.3 環境変数ファイルを設定する

> **「環境変数」とは？**
> アプリの動作を変えるための設定値です。
> パスワードや URL など、コードに直接書きたくない情報を外部ファイルで管理します。
> `.env` ファイルは Git に含まれないため、各自で作成する必要があります。

プロジェクトのルートフォルダ（`mail-dispatcher/` 直下）に `.env` ファイルを**新規作成**し、以下の内容を記述してください。

```env
# データベース設定
# SQLite のデータベースファイルのパス（開発環境はこのまま使用）
DATABASE_URL="file:./prisma/dev.db"

# アプリケーション設定
# アプリのベース URL（開発環境はこのまま）
APP_URL="http://localhost:3000"

# セッション署名用シークレットキー（64文字以上のランダムな文字列を設定）
SESSION_SECRET="ここに64文字以上のランダムな文字列を貼り付ける"

# メール送信モード
# console = ターミナルに表示するだけ（実際には送信しない）← 研修ではこちらを使用
# smtp    = 実際にメールを送信する
EMAIL_MODE=console

# SMTP設定（EMAIL_MODE=smtp の場合のみ必要）
# 研修環境では EMAIL_MODE=console のため、以下は設定不要
SMTP_HOST=smtp.example.com
SMTP_PORT=587
SMTP_USER=your_user
SMTP_PASS=your_pass
MAIL_FROM=No Reply <no-reply@example.com>
```

#### SESSION_SECRET の生成方法

以下のコマンドでランダムな文字列を生成できます。

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

出力例:
```
a3f8b2c1d9e4f7a2b5c8d3e6f1a4b7c0d3e6f9a2b5c8d1e4f7a0b3c6d9e2f5a8
```

この文字列を `SESSION_SECRET` の値として設定してください。

> **なぜ SESSION_SECRET が必要か？**
> ログイン状態を管理する Cookie の改ざんを防ぐための秘密鍵です。
> 推測されると悪意のある人がログイン状態を偽造できるため、ランダムな文字列を使います。

---

### 2.4 データベースをセットアップする

> **「マイグレーション」とは？**
> `prisma/schema.prisma`（テーブルの設計図）をデータベースに反映する作業です。

**手順1: マイグレーション実行（テーブル作成）**

```bash
npm run prisma:migrate
```

実行後、`prisma/dev.db`（SQLite データベースファイル）が自動的に作成されます。

実行例:
```
SQLite database dev.db created at file:./prisma/dev.db

The following migration(s) have been applied:
  migrations/
    20260101000000_init/
      migration.sql

Your database is now in sync with your schema.
```

**手順2: Prisma クライアント生成**

```bash
npm run prisma:generate
```

> **なぜこの手順が必要か？**
> `schema.prisma` をもとに TypeScript の型定義を生成します。
> これにより、テーブルのカラム名やデータ型の間違いをコード上で事前に検出できます。

実行すると以下のデータが作成されます。

## 3. 初回ログイン手順

シード直後はパスワードが設定されていないため、**Magic Link** でログインします。

### Magic Link とは

Magic Link はパスワード不要でログインできる仕組みです。
開発環境では実際のメールは送信されず、**ターミナルに URL が表示されます**。

ログインの流れ:

1. アプリのログイン画面でメールアドレスを入力して「Magic Link を送る」をクリック
2. `npm run dev` を実行したターミナルに URL が表示される
3. その URL をコピーしてブラウザに貼り付けてアクセスする
4. 自動的にダッシュボードへ遷移してログイン完了

### 3.1 Magic Link でログインする手順

1. ブラウザで `http://localhost:3000/login` を開く

2. 「Magic Link」タブをクリックする

3. 管理者のメールアドレスを入力する
   ```
   shige03nori@gmail.com
   ```

4. 「Magic Link を送る」ボタンをクリックする

5. **`npm run dev` を実行したターミナル**を確認する。以下のような出力が表示される:
   ```
   ========================================
   Magic Link (DEV)
   To: shige03nori@gmail.com
   URL: http://localhost:3000/api/auth/verify?token=xxxxxx
   ========================================
   ```

6. 表示された URL をコピーして、ブラウザのアドレスバーに貼り付けてアクセスする

7. ダッシュボード画面に遷移すればログイン完了

> **注意: Magic Link の有効期限は 15 分です**
> 15分を過ぎると「リンクが無効です」エラーになります。
> その場合はもう一度「Magic Link を送る」を実行してください。

### 3.2 パスワードを設定する（推奨）

毎回 Magic Link を使うのは手間なので、パスワードを設定しておくと便利です。

1. ダッシュボードの「ユーザー管理」を開く（ADMIN のみ）
2. 対象ユーザーの「パスワード設定」ボタンをクリックする
3. 新しいパスワード（8文字以上）を入力して保存する

以降は以下の手順でログインできます:
1. `http://localhost:3000/login` を開く
2. 「パスワード」タブを選択（デフォルト）
3. メールアドレスとパスワードを入力してログイン

## 4. よく使うコマンド

| コマンド | 説明 | いつ使う |
|---------|------|---------|
| `npm run dev` | 開発サーバー起動 | 毎日の開発開始時 |
| `npm run build` | 本番用ビルド | 本番環境にデプロイする前 |
| `npm run start` | 本番用サーバー起動 | `build` 後に実行 |
| `npm run prisma:migrate` | DB マイグレーション実行 | schema.prisma を変更した後 |
| `npm run prisma:generate` | Prisma クライアント再生成 | schema.prisma を変更した後 |

| `npm run studio` | Prisma Studio 起動 | DB の中身を GUI で確認したい時 |
| `npm run dev:full` | フルリセット + 起動 | schema 変更後に全て一括実行したい時 |

## 5. スキーマ変更後の手順

`prisma/schema.prisma` を編集した場合は以下の手順を踏んでください。

```bash
# 1. マイグレーション実行（DB にテーブル変更を反映）
npm run prisma:migrate
# ※ マイグレーション名の入力を求められたら英語で入力
#    例: add_note_field、create_tags_table など

# 2. Prisma クライアント再生成（TypeScript の型定義を更新）
npm run prisma:generate

# 3. 開発サーバーを再起動
# Ctrl+C でサーバーを止めてから:
npm run dev
```

一括実行したい場合:

```bash
npm run dev:full
```

## 6. Prisma Studio（DB GUI）

> **「Prisma Studio」とは？**
> データベースの中身をブラウザで確認・編集できる GUI ツールです。
> Excel のような表形式でデータを閲覧・追加・編集・削除できます。

```bash
npm run studio
```

実行後、ブラウザで `http://localhost:5555` を開くと Prisma Studio が表示されます。

| 操作 | 説明 |
|------|------|
| データ閲覧 | テーブル内のデータを一覧表示 |
| データ追加 | レコードを手動で追加 |
| データ編集 | セルをクリックして直接編集 |
| データ削除 | レコードを削除 |
| フィルター | 条件でデータを絞り込み |

> **Prisma Studio は開発・デバッグ用のツールです。本番環境では使用しないでください。**

## 7. トラブルシューティング

### 問題1: `npm install` が失敗する

確認事項:
- `node -v` で v20 以上が表示されるか
- インターネット接続が有効か
- `mail-dispatcher` フォルダ内にいるか（`ls package.json` でファイルが表示されれば OK）

---

### 問題2: `npm run prisma:migrate` が失敗する

**症状**: 「migration failed」または「Database error」が表示される

**原因**: 既存の `prisma/dev.db` が壊れている可能性があります。

**解決策**（開発環境のみ）:

```bash
# ⚠️ データがすべて消えるため注意
rm prisma/dev.db

# DB を再作成してシードも再実行
npm run prisma:migrate
npm run seed
```

または一括で:

```bash
npm run dev:full
```

---

### 問題3: Magic Link の URL がターミナルに表示されない

確認手順:

1. **ターミナルを正しく見ているか確認**
   - ブラウザではなく、`npm run dev` を実行したターミナルを見る
   - ターミナルが複数開いている場合は `npm run dev` を実行したものを確認

2. **Magic Link の有効期限（15分）が切れていないか確認**
   - 切れていた場合は再度「Magic Link を送る」を実行する

3. **メールアドレスが正しいか確認**
   - `shige03nori@gmail.com` を正確に入力しているか確認する

---

### 問題4: `npm run seed` でエラーが出る

```bash
# Prisma クライアントが古い場合はこれを試す
npm run prisma:generate
npm run seed
```

それでも解決しない場合:

```bash
npm run prisma:migrate
npm run prisma:generate
npm run seed
```

---

### 問題5: `http://localhost:3000` が開かない

確認手順:

1. 開発サーバーが起動しているか確認
   - ターミナルに `Ready in X.Xs` が表示されているか確認

2. ポート 3000 が既に使われていないか確認

```bash
# Windows（Git Bash）
netstat -ano | findstr :3000
```

ポートが使用中の場合は実行中の別の `npm run dev` を停止する。

---

### 問題6: Windows で `npm run seed` が動かない

**解決策1 - Git Bash を使う（推奨）**:
```bash
npm run seed
```

**解決策2 - PowerShell の実行ポリシーを変更する**:
```powershell
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
```

---

### 問題7: Prisma Studio が開かない

- 自動でブラウザが開かない場合は手動で `http://localhost:5555` を開く
- ポート 5555 が使用中の場合は、他の Prisma Studio プロセスを終了する
