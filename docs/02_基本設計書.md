## 3. データモデル概要

### 3.1 エンティティ関連図（概略）

```
Organization（組織）
  ├── Membership（メンバーシップ） → User（ユーザー）
  ├── Invitation（招待）
  ├── Contact（連絡先）
  │     └── groups: JSON   → ContactGroup（グループ）参照
  ├── ContactGroup（グループ）
  ├── EmailTemplate（メールテンプレート）
  └── EmailCampaign（キャンペーン）
        └── EmailCampaignRecipient（配信先明細） → Contact
```

### 3.2 主要エンティティ

| エンティティ | 役割 |
|------------|------|
| Organization | テナント単位。全データはこれに紐づく |
| User | ログインユーザー。複数の組織に所属可能 |
| Membership | User と Organization の紐づけ＋ロール |
| Contact | 配信先の連絡先情報 |
| ContactGroup | 連絡先のグループ分類ラベル |
| EmailTemplate | 再利用可能なメールのひな型 |
| EmailCampaign | 1回の配信バッチ（送信実績のスナップショットを保持） |
| EmailCampaignRecipient | キャンペーンの宛先ごとの送信結果明細 |
| Invitation | ユーザー招待トークン |
| LoginToken | マジックリンク用ワンタイムトークン |

**成功時**

```json

```

## 5. 認証・認可設計

### 5.1 認証方式

| 方式 | 概要 |
|------|------|
| パスワード認証 | メールアドレス＋パスワード（bcrypt ハッシュと照合） |
| マジックリンク | メール送信されたワンタイム URL。トークンはハッシュ化して DB 保存 |

### 5.2 セッション管理

- サーバーサイドで Cookie（`session` キー）を発行
- Cookie は `HttpOnly`・`SameSite` 設定でセキュアに保持
- ページアクセス・API アクセス時にセッションを検証

### 5.3 認可（ロールベースアクセス制御）

```
VIEWER  → GET 系のみ許可
EDITOR  → GET + 作成・更新・削除（ユーザー管理系を除く）
ADMIN   → 全操作（ユーザー管理・パスワード設定を含む）
```

- API Route 側でロールチェックを実施し、不正な場合は 403 を返す
- ページ側でも権限に応じてボタン・リンクの表示/非表示を制御する

## 6. メール送信設計

### 6.1 送信フロー

```
1. ユーザーが送信対象の連絡先を選択
2. メール作成画面でテンプレート選択または直接入力
3. 送信ボタン押下（Server Action 実行）
4. EmailCampaign レコード作成（status: SENDING）
5. EmailCampaignRecipient を宛先分まとめて作成
6. 各宛先に対してループ送信
   ├── メールアドレスなし → SKIPPED
   ├── 送信成功       → SENT ＋ messageId 保存
   └── 送信失敗       → FAILED ＋ エラーメッセージ保存
7. EmailCampaign の集計カウントと status を確定更新
8. 結果画面にリダイレクト
```

### 6.2 差し込み変数

| 変数 | 置換元 |
|------|--------|
| `{{name}}` | Contact.name |
| `{{companyName}}` | Contact.companyName |
| `{{email}}` | Contact.email |
| `{{phone}}` | Contact.phone |

### 6.3 スナップショット保存

送信時点の件名・本文は `EmailCampaign` にスナップショットとして保存する（テンプレート編集後も送信時の内容を再現できる）。

### 6.4 メール送信モード

環境変数 `EMAIL_MODE` でメール送信の動作を切り替える。

| `EMAIL_MODE` の値 | 動作 |
|------------------|------|
| `console`（デフォルト） | メール内容をサーバーのターミナルに出力する。実際には送信しない |
| `smtp` | Nodemailer を使って SMTP サーバーに実際に送信する |

**コンソール出力形式（`EMAIL_MODE=console`）**

```
=== EMAIL (console) ===
to: <宛先メールアドレス>
subject: <件名>
text: <本文テキスト>
html: <HTML本文（存在する場合）>
=======================
```

**SMTP 設定に必要な環境変数（`EMAIL_MODE=smtp`）**

| 変数名 | 説明 |
|--------|------|
| `SMTP_HOST` | SMTP サーバーのホスト名 |
| `SMTP_PORT` | SMTP ポート番号 |
| `SMTP_USER` | SMTP 認証ユーザー名 |
| `SMTP_PASS` | SMTP 認証パスワード |
| `MAIL_FROM` | 送信元メールアドレス |

**マジックリンクのメール通知**

マジックリンクのメール送信は `EMAIL_MODE` に関わらず、常にターミナルにコンソール出力する（`src/lib/mail.ts`）。

```
========================================
📩 Magic Link (DEV)
To: <宛先メールアドレス>
URL: <ワンタイムログインURL>
========================================
```

## 7. データ保全・制約

| 項目 | 設計 |
|------|------|
| 組織削除 | `onDelete: Cascade` で関連データをすべて削除 |
| ユーザー削除 | 連絡先・キャンペーンの createdBy は `onDelete: Restrict`（削除不可） |
| 連絡先削除 | 送信先明細（Recipient）の contactId は `SetNull`（履歴は保持） |
| テンプレート削除（アーカイブ） | 物理削除でなく `isArchived: true` フラグで管理 |
| メールアドレス重複 | 同組織内で一意制約。API 側で 409 を返す |
| グループ名重複 | 同組織内で一意制約。API 側で `name_already_exists` エラーを返す |
