## 1. システム構成

### 1.1 アーキテクチャ概要

> **「アーキテクチャ」とは？**
> システムの「骨組み」のことです。どんな部品をどのように組み合わせてシステムを作るかの設計図です。

**システムの構成要素と流れ**

```
[クライアント] ブラウザ（Chrome / Edge など）
      |
      | HTTP(S) リクエスト / レスポンス
      v
[サーバー] Next.js アプリ
      |
      +-- Server Components  : ページの HTML を生成。DB に直接アクセス
      +-- Client Components  : ブラウザ側で動く UI（ボタン・フォームなど）
      +-- Server Actions     : フォーム送信処理。ページ遷移（リダイレクト）も担当
      +-- API Routes         : /api/... の URL で呼び出せる REST API
      |
      | Prisma ORM（SQL を書かずに DB を操作）
      v
[データ層] SQLite データベース（prisma/dev.db）
      ファイル 1 つで完結する DB。研修環境向け。

      |
      | Nodemailer（メール送信ライブラリ）
      v
[メール送信]
      +-- console モード: ターミナルに出力するだけ（実際には送信しない）
      +-- smtp モード  : SMTP サーバーに実際に送信する
```

> **用語解説**
> - **Server Components**: サーバー側で HTML を生成するコンポーネント
> - **Client Components**: ブラウザ側で動くコンポーネント。ボタンのクリック処理などを担当
> - **Server Actions**: フォーム送信などを処理するサーバー側の関数
> - **API Routes**: `/api/...` の URL で呼び出せるサーバー処理。JSON でデータをやり取りする
> - **Prisma ORM**: SQL を書かずに JavaScript/TypeScript で DB 操作できるライブラリ

## 3. データモデル概要

> **「データモデル」とは？**
> データベースにどんな情報を保存するかの設計図です。

### 3.1 エンティティ関連図

```
Organization（組織）
  |
  +-- Membership（メンバーシップ）--- role ---> ADMIN / EDITOR / VIEWER
  |         |
  |         +-- User（ユーザー）
  |               |
  |               +-- LoginToken（Magic Link トークン。有効期限 15 分）
  |
  +-- Invitation（招待。有効期限 72 時間）
  |
  +-- Contact（連絡先）
  |       groups フィールド --[JSON 配列]--> ContactGroup の ID を参照
  |
  +-- ContactGroup（グループ）
  |
  +-- EmailTemplate（テンプレート）
  |       isArchived: true でアーカイブ（物理削除しない）
  |
  +-- EmailCampaign（配信キャンペーン）
          subjectSnapshot  : 送信時の件名コピー
          textBodySnapshot : 送信時の本文コピー
          status           : DRAFT / SCHEDULED / SENDING / SENT / FAILED
          |
          +-- EmailCampaignRecipient（宛先明細）
                  emailSnapshot : 送信時のメールアドレスコピー
                  status        : PENDING / SENT / FAILED / SKIPPED
                  contactId     : Contact へのリンク（削除後は null）
```

### 3.2 主要エンティティの説明

| エンティティ | 現実のイメージ | 役割 |
|------------|-------------|------|
| Organization | 会社・チーム | テナント単位。全データはこれに紐づく |
| User | ログインするアカウント | メールアドレスで一意。複数の組織に所属可能 |
| Membership | 社員証 | User と Organization の紐づけ＋ロール情報 |
| Contact | 名刺 | 配信先の連絡先情報 |
| ContactGroup | 名刺のラベル | 連絡先のグループ分類 |
| EmailTemplate | メールのひな型 | 再利用可能なメールのフォーマット |
| EmailCampaign | 送信バッチ | 1回の配信単位。件名・本文のスナップショットを保持 |
| EmailCampaignRecipient | 送付状の控え | キャンペーンの宛先ごとの送信結果明細 |
| Invitation | 招待状 | ユーザー招待用のワンタイムトークン（72時間有効） |
| LoginToken | Magic Link の実体 | ワンタイムログイン用トークン（15分有効） |

> **「スナップショット」とは？**
> 送信時点の内容をそのままコピーして保存しておく仕組みです。
> テンプレートを後から編集しても、過去の送信内容は変わらず確認できます。

## 5. 認証・認可設計

> **「認証」と「認可」の違い**
> - **認証（Authentication）**: 「あなたは誰ですか？」→ ログイン処理
> - **認可（Authorization）**: 「あなたはこれをしていいですか？」→ 権限チェック

### 5.1 認証方式

**パスワード認証**

1. メールアドレスとパスワードを入力
2. DB から passwordHash を取得
3. bcrypt でパスワードを照合
4. 一致した場合のみセッション発行 → ダッシュボードへ

**Magic Link 認証**

1. メールアドレスを入力
2. ランダムトークンを生成し、ハッシュ化して DB に保存（有効期限 15 分）
3. ターミナルに URL を出力
4. ユーザーが URL をブラウザで開く
5. トークンをハッシュ化して DB と照合。有効期限・使用済みチェック
6. セッション発行 → ダッシュボードへ

| 方式 | 概要 | 有効期限 |
|------|------|---------|
| パスワード認証 | メールアドレス＋パスワード（bcrypt ハッシュと照合） | — |
| Magic Link | ターミナルに表示されるワンタイム URL | 15 分 |

### 5.2 セッション管理

- ログイン成功後、`{ userId, organizationId, role }` を JSON 化して Cookie に保存
- Cookie は `HttpOnly: true`（JavaScript からアクセス不可）・`SameSite: lax` でセキュアに保持
- 以降のすべてのリクエストで Cookie を読み取ってセッションを検証

### 5.3 認可の流れ

1. セッションなし → **401 Unauthorized**（未認証）
2. VIEWER が変更操作をしようとした → **403 Forbidden**（権限不足）
3. 他組織のデータへのアクセス → **404 Not Found**
4. 通過 → 処理実行

## 6. メール送信設計

### 6.1 即時送信フロー

1. ユーザーが送信ボタンをクリック
2. EmailCampaign を DB に作成（status: `SENDING`）
3. EmailCampaignRecipient を宛先分まとめて作成
   - メールアドレスあり → `PENDING`
   - メールアドレスなし → `SKIPPED`
4. 結果ページにリダイレクト（3秒ごとに自動更新）
5. バックグラウンドで `processCampaign()` が各宛先にメール送信
   - 差し込み変数を置換してから送信
   - 成功 → `SENT`、失敗 → `FAILED`
6. 全件完了後、Campaign のカウントと status を更新

### 6.2 予約送信フロー

1. ユーザーが送信日時を指定して送信ボタンをクリック
2. EmailCampaign を作成（status: `SCHEDULED`、scheduledAt を保存）
3. 結果ページにリダイレクト（60秒ごとに自動更新）
4. `scheduler.ts` が 60 秒ごとにポーリング
5. scheduledAt を過ぎた SCHEDULED キャンペーンを検知 → status を `SENDING` に更新
6. `processCampaign()` でバックグラウンド送信処理

### 6.3 差し込み変数

| 変数 | 置換される値 | 例 |
|------|------------|-----|
| `{{name}}` | 連絡先の氏名 | 山田 太郎 |
| `{{companyName}}` | 連絡先の会社名 | 株式会社サンプル |
| `{{email}}` | 連絡先のメールアドレス | yamada@example.com |
| `{{phone}}` | 連絡先の電話番号 | 090-0000-0000 |

### 6.4 メール送信モード切り替え

環境変数 `EMAIL_MODE` で動作を切り替える。

| `EMAIL_MODE` | 動作 | 用途 |
|-------------|------|------|
| `console`（デフォルト） | ターミナルにメール内容を出力するだけ。実際には送信しない | 研修・開発環境 |
| `smtp` | SMTP サーバーに実際に送信する。別途 SMTP_HOST 等の設定が必要 | 本番環境 |

コンソール出力例:
```
=== EMAIL (console) ===
to: yamada@example.com
subject: 山田 太郎 様 〜ご案内〜
text: （本文）
=======================
```

## 7. データ保全・制約

| 項目 | 設計 | 理由 |
|------|------|------|
| 組織削除 | `onDelete: Cascade` で関連データをすべて削除 | 孤立データを残さない |
| ユーザー削除 | `onDelete: Restrict`（削除不可） | 過去の操作者情報を保全する |
| 連絡先削除 | 送信先明細の contactId は `SetNull`（履歴は保持） | 削除後も送信履歴が確認できる |
| テンプレート削除 | 物理削除でなく `isArchived: true` フラグで管理 | 過去の配信履歴との紐づけを保持 |
| メールアドレス重複 | 同組織内で一意制約。API 側で 409 を返す | 同じ相手への二重登録を防ぐ |
| グループ名重複 | 同組織内で一意制約。API 側でエラーを返す | 混乱を防ぐ |
