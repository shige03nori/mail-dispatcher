## 1. データベース詳細設計

### テーブル関係の全体像

```
Organization（組織）
  |
  +-- Membership（メンバーシップ）
  |       organizationId + userId で一意
  |       role: ADMIN / EDITOR / VIEWER
  |         |
  |         +-- User（ユーザー）
  |               email で全体一意
  |               passwordHash: null の場合は Magic Link のみでログイン可
  |               status: INVITED / ACTIVE / DISABLED
  |               |
  |               +-- LoginToken（Magic Link トークン）
  |                       有効期限 15 分、使用後は usedAt が設定される
  |
  +-- Invitation（招待）
  |       有効期限 72 時間、使用後は acceptedAt が設定される
  |
  +-- Contact（連絡先）
  |       groups フィールド: ContactGroup の ID を JSON 配列で保存
  |       例: '["id1", "id2"]'
  |
  +-- ContactGroup（グループ）
  |       name は組織内で一意
  |
  +-- EmailTemplate（テンプレート）
  |       name は組織内で一意
  |       isArchived: true で論理削除（物理削除しない）
  |
  +-- EmailCampaign（配信キャンペーン）
          送信時の件名・本文をスナップショットとして保持
          status: DRAFT / SCHEDULED / SENDING / SENT / FAILED
          |
          +-- EmailCampaignRecipient（宛先明細）
                  contactId: 連絡先削除後は null になる
                  emailSnapshot: 送信時のメールアドレスを保持
                  status: PENDING / SENT / FAILED / SKIPPED
```

## 2. API 詳細設計

### 2.1 認証 API

#### POST `/api/auth/login` — パスワード認証

リクエスト:
```json
{ "email": "user@example.com", "password": "password123" }
```

処理の流れ:

1. IP 別リクエスト数チェック（15分間に10回超）→ 超過なら 429 Too Many Requests
2. email で User を DB 検索 → 見つからない場合 401 `invalid_credentials`
3. passwordHash が null（未設定）→ 401 `no_password`（Magic Link でログインが必要）
4. bcrypt でパスワードを照合 → 不一致なら 401 `invalid_credentials`
5. Membership を取得（organizationId, role を確認）
6. Cookie にセッション情報を保存 → 200 `{ ok: true }` → `/dashboard` へ

---

#### POST `/api/auth/request-link` — Magic Link 送信

リクエスト:
```json
{ "email": "user@example.com" }
```

処理の流れ:

1. email で User を DB 検索
   - **ユーザーが存在しなくても `{ ok: true }` を返す**（セキュリティ対策）
2. ランダムトークンを生成（32バイト）→ SHA-256 でハッシュ化
3. DB に保存（有効期限: 15分後）
4. ターミナルに出力:

```
========================================
Magic Link (DEV)
To: user@example.com
URL: http://localhost:3000/api/auth/verify?token=xxxxx
========================================
```

---

#### GET `/api/auth/verify?token=<token>` — Magic Link 検証

処理の流れ:

1. token を SHA-256 でハッシュ化して DB から検索
2. 存在しない → `/login?err=invalid` へリダイレクト
3. 有効期限切れ（15分経過）→ `/login?err=invalid` へリダイレクト
4. 使用済み（usedAt が null でない）→ `/login?err=invalid` へリダイレクト
5. `usedAt` を現在時刻で更新（2回目の使用を防ぐ）
6. セッション発行 → `/dashboard` へリダイレクト

---

#### POST `/api/auth/logout` — ログアウト

セッション Cookie を削除して `/login` へリダイレクト。

---

#### POST `/api/auth/password` — 自分のパスワード変更

リクエスト:
```json
{ "password": "newpassword123" }
```

| チェック | エラー |
|--------|-------|
| 未認証 | 401 Unauthorized |
| 8 文字未満 | 400 `password_too_short` |

処理: bcrypt でハッシュ化 → `User.passwordHash` を更新 → `{ ok: true }`

---

### 2.2 連絡先 API

#### GET `/api/contacts?groupId=<id>` — 連絡先一覧取得

処理の流れ:

1. セッション確認 → 未認証なら 401
2. `organizationId` スコープで全件取得（createdAt 昇順）
3. `groups` フィールドを JSON.parse で配列に変換
4. `groupId` が指定されている場合、groups 配列に groupId を含む連絡先のみ絞り込む

レスポンス例:
```json
{
  "ok": true,
  "contacts": [
    {
      "id": "clxxx",
      "name": "山田 太郎",
      "companyName": "株式会社サンプル",
      "email": "yamada@example.com",
      "phone": "090-0000-0000",
      "note": "メモ",
      "groups": ["groupId1", "groupId2"],
      "createdAt": "2026-01-01T00:00:00.000Z"
    }
  ]
}
```

---

#### POST `/api/contacts` — 連絡先新規作成

リクエスト:
```json
{
  "name": "山田 太郎",
  "companyName": "株式会社サンプル",
  "email": "yamada@example.com",
  "phone": "090-0000-0000",
  "note": "メモ",
  "groupIds": ["groupId1"]
}
```

| チェック | エラー |
|--------|-------|
| `name` が空 | 400 `name_required` |
| `email` が同組織内で重複 | 409 `email_already_exists` |
| VIEWER ロール | 403 `forbidden` |
| Zod スキーマ不合格 | 400 `validation_error` |

---

#### PUT `/api/contacts/[id]` — 連絡先更新

リクエスト形式は POST と同じ。

処理の流れ:
1. id + organizationId で対象を検索（存在しない → 404）
2. 全フィールドを更新（groups は JSON.stringify で文字列化）
3. updatedByUserId を現在のユーザー ID に更新

---

#### DELETE `/api/contacts/[id]` — 連絡先削除

処理の流れ:
1. id + organizationId で対象を検索（存在しない → 404）
2. 削除実行
3. EmailCampaignRecipient.contactId は SetNull（送信履歴は残る）

---

#### GET `/api/contacts/export` — CSV エクスポート

- Content-Type: `text/csv; charset=utf-8`
- BOM 付き UTF-8（Excel で文字化けしないよう対応）
- ヘッダー行: `氏名,メール,電話,会社名,メモ,グループ`

---

#### POST `/api/contacts/import` — CSV インポート

処理の流れ:

1. CSV ファイルを受け取る
2. 1行ずつパース
   - name が空の行 → スキップ
   - email が同組織内で重複する行 → スキップ（skipped++）
   - それ以外 → インポート対象に追加
3. 一括 createMany で DB に保存

レスポンス:
```json
{ "ok": true, "imported": 5, "skipped": 2 }
```

## 3. Server Actions 詳細設計

> **「Server Actions」とは？**
> フォームの送信をサーバー側で処理する Next.js の機能です。
> ページ遷移（リダイレクト）を含む操作に使います。

### 3.1 テンプレート作成 (`createTemplateAction`)

入力: `FormData`（name, subject, textBody, htmlBody）

処理の流れ:
1. セッション確認 → VIEWER なら `/dashboard/templates?err=forbidden` へリダイレクト
2. Zod スキーマ検証
3. `EmailTemplate.create()` で DB に保存
4. `/dashboard/templates` へリダイレクト

### 3.2 テンプレート更新 (`updateTemplateAction`)

処理の流れ:
1. セッション確認（VIEWER → forbidden リダイレクト）
2. organizationId + templateId で存在確認（存在しない → 404）
3. `EmailTemplate.update` で全フィールド更新
4. `/dashboard/templates/[id]/edit?ok=1` へリダイレクト

### 3.3 テンプレートアーカイブ / 復元

| Action | 処理 | リダイレクト先 |
|--------|------|-------------|
| `archiveTemplateAction` | `isArchived = true` に更新 | /dashboard/templates |
| `restoreTemplateAction` | `isArchived = false` に更新 | /dashboard/templates |

## 4. メール送信処理 詳細設計

### 4.1 メール送信モード切り替え

`src/lib/email/index.ts` が `EMAIL_MODE` 環境変数を参照して実装を切り替える。

- `EMAIL_MODE=console` → `sender.console.ts` を使用
  - ターミナルにメール内容を出力するだけ
  - messageId は `"console-<timestamp>"` を返す
- `EMAIL_MODE=smtp` → `sender.smtp.ts` を使用
  - Nodemailer で SMTP サーバーに実際に送信
  - 必要な環境変数: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `MAIL_FROM`

---

### 4.2 メール配信 Server Action（`sendAction`）

入力: `FormData`（ids, templateId, subject, textBody, htmlBody, scheduledAt）

処理の流れ:

1. セッション確認・ロールチェック → VIEWER なら compose 画面にリダイレクト
2. ids をパース（カンマ区切り文字列 → 配列、最大 500 件・重複除去）
3. バリデーション
   - subject が空 → エラーリダイレクト
   - textBody が空 → エラーリダイレクト
4. scheduledAt をパース
   - 未来の日時 → `isScheduled = true`（予約送信）
   - 空または過去 → `isScheduled = false`（即時送信）
5. DB から Contact を再取得（URL 改ざん対策）
6. EmailCampaign を作成
   - `isScheduled=true` → status: `SCHEDULED`, scheduledAt 保存
   - `isScheduled=false` → status: `SENDING`
7. EmailCampaignRecipient を `createMany` で一括作成
   - email あり → status: `PENDING`
   - email なし → status: `SKIPPED`, errorMessage: `"email is missing"`
8. `isScheduled=false` の場合のみ、`setImmediate` で `processCampaign()` をバックグラウンド実行
9. `/dashboard/compose/result?campaignId=xxx` へリダイレクト

---

### 4.3 キャンペーン処理関数（`processCampaign`）

`src/lib/email/processCampaign.ts` に切り出した送信ロジック。
`sendAction` と `scheduler.ts` の双方から呼び出される。

入力: `campaignId: string`

処理の流れ:

1. EmailCampaign を取得（textBodySnapshot, htmlBodySnapshot など）
2. status が `PENDING` の EmailCampaignRecipient を取得
3. 各 Recipient をループ処理
   - contactId で Contact を取得（name, companyName など）
   - 差し込み変数を置換（`applyVars` 関数）
   - `sendEmail()` を呼び出し
   - 成功 → status: `SENT`, providerMessageId を保存
   - 失敗 → status: `FAILED`, errorMessage を保存
4. EmailCampaign を集計更新
   - sentCount, failedCount, skippedCount, totalCount を確定
   - `failedCount > 0` → status: `FAILED`
   - `failedCount = 0` → status: `SENT`

**差し込み変数置換関数（applyVars）**

```typescript
function applyVars(template: string, contact: Contact): string {
  return template
    .replaceAll("{{name}}", contact.name ?? "")
    .replaceAll("{{companyName}}", contact.companyName ?? "")
    .replaceAll("{{email}}", contact.email ?? "")
    .replaceAll("{{phone}}", contact.phone ?? "");
}
```

> `?? ""` は、値が null / undefined の場合に空文字列に置き換える記法です。

---

### 4.4 予約送信スケジューラ（`scheduler.ts`）

`src/instrumentation.ts` フックから起動する Node.js 常駐プロセス。

起動条件: `process.env.NEXT_RUNTIME === "nodejs"`（Edge Runtime では実行しない）

動作:
- サーバー起動時に即時実行 → その後 60 秒ごとにポーリング

`runScheduledCampaigns()` の処理:

1. `status=SCHEDULED` かつ `scheduledAt <= 現在時刻` のキャンペーンを検索
2. 各キャンペーンに対して:
   - `updateMany({ where: { id, status: SCHEDULED } })` で status を `SENDING` に更新
   - `count = 0` → 他プロセスが先取りしたためスキップ（二重処理防止）
   - `count = 1` → `processCampaign(id)` をバックグラウンド実行

> **二重処理防止の仕組み**
> 複数のプロセスが同時に動いていても、`updateMany` の戻り値（count）が 0 であれば
> 他のプロセスが先に処理したことがわかり、スキップできる。

## 5. セッション詳細設計

### セッション構造

```typescript
type Session = {
  userId: string;          // ログインユーザーの ID
  organizationId: string;  // 所属組織の ID
  role: "ADMIN" | "EDITOR" | "VIEWER";  // 権限
}
```

### セッションの取得（`getSession`）

1. Cookie から `session` を取得 → Cookie なし → null を返す
2. JSON パース
3. 必須フィールド（userId / organizationId / role）が揃っているか確認
   - 足りない → null を返す
   - OK → Session オブジェクトを返す

### セッションの発行（`createSession`）

1. Session オブジェクトを JSON 文字列化
2. `HttpOnly: true`, `SameSite: lax`, `Path: /` で Cookie にセット

## 6. エラーコード一覧

> **HTTP ステータスコードとは？**
> - 200 番台: 成功
> - 400 番台: クライアント（ユーザー側）のエラー
> - 500 番台: サーバー側のエラー

| コード | HTTP | 説明 |
|--------|------|------|
| `unauthorized` | 401 | 未認証（ログインが必要） |
| `forbidden` | 403 | 権限不足（このロールでは操作できない） |
| `not_found` | 404 | 対象リソースが存在しない（または他組織のデータ） |
| `name_required` | 400 | 名前が空 |
| `email_already_exists` | 409 | メールアドレスが同組織内で重複 |
| `name_already_exists` | 409 | グループ名が同組織内で重複 |
| `no_password` | 401 | パスワード未設定（Magic Link でログインが必要） |
| `invalid_credentials` | 401 | パスワード不一致 |
| `invalid_token` | 400 | トークンが無効・期限切れ・使用済み |
| `password_too_short` | 400 | パスワードが 8 文字未満 |
| `validation_error` | 400 | Zod スキーマバリデーション失敗 |
| `subject_required` | 400 | 件名が空（compose バリデーション） |
| `body_required` | 400 | 本文（textBody）が空（compose バリデーション） |
