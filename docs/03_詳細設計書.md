## 2. API 詳細設計

### 2.1 認証 API

#### POST `/api/auth/login`

パスワード認証を行い、セッション Cookie を発行する。

**リクエスト**
```json
{ "email": "user@example.com", "password": "password123" }
```

**処理フロー**
1. email で User を検索
2. `passwordHash` が null → `{ ok: false, error: "no_password" }`
3. bcrypt で照合 → 不一致 → `{ ok: false, error: "invalid_credentials" }`
4. Membership を取得しセッション発行（organizationId, userId, role を格納）
5. `{ ok: true }` を返し `/dashboard` にリダイレクト

**レスポンス（成功）**
```json
{ "ok": true }
```

---

#### POST `/api/auth/request-link`

マジックリンクを発行してメール送信する。

**リクエスト**
```json
{ "email": "user@example.com" }
```

**処理フロー**
1. email で User を検索（存在しない場合でも `{ ok: true }` を返す）
2. ランダムトークン生成 → SHA-256 ハッシュ化して DB 保存（有効期限 15分）
3. `GET /api/auth/verify?token=<rawToken>` の URL を生成
4. Nodemailer でメール送信（開発環境ではコンソール出力）
5. `{ ok: true }` を返す

---

#### GET `/api/auth/verify?token=<token>`

マジックリンクを検証してセッション発行する。

**処理フロー**
1. `token` を SHA-256 ハッシュ化して DB から検索
2. 存在しない・有効期限切れ・使用済み → `/login?err=invalid` にリダイレクト
3. `usedAt` を現在時刻で更新
4. セッション発行 → `/dashboard` にリダイレクト

---

#### POST `/api/auth/logout`

セッション Cookie を削除してログアウトする。

---

### 2.2 連絡先 API

#### GET `/api/contacts?groupId=<id>`

組織スコープの連絡先一覧を返す。

| パラメータ | 必須 | 説明 |
|-----------|------|------|
| groupId | — | 指定時はそのグループの連絡先のみ返す |

**処理フロー**
1. セッション検証（未認証 → 401）
2. `organizationId` スコープで全件取得（orderBy: createdAt asc）
3. `groups` JSON をパース
4. `groupId` が指定されていれば絞り込み

**レスポンス**
```json
{
  "ok": true,
  "contacts": [
    {
      "id": "...",
      "name": "山田 太郎",
      "companyName": "株式会社サンプル",
      "email": "yamada@example.com",
      "phone": "090-0000-0000",
      "note": "メモ",
      "groups": ["groupId1", "groupId2"],
      "createdAt": "2026-01-01T00:00:00.000Z"
    }
  ]
}
```

---

#### POST `/api/contacts`

連絡先を新規作成する。

**リクエスト**
```json
{
  "name": "山田 太郎",
  "companyName": "株式会社サンプル",
  "email": "yamada@example.com",
  "phone": "090-0000-0000",
  "note": "メモ",
  "groupIds": ["groupId1"]
}
```

**バリデーション**
- `name` が空 → 400 `name_required`
- `email` が同組織内で重複 → 409 `email_already_exists`
- VIEWER ロール → 403 `forbidden`

---

#### PUT `/api/contacts/[id]`

連絡先を更新する。リクエスト形式は POST と同じ。

**処理フロー**
1. `id` と `organizationId` で対象を検索（存在しない → 404）
2. 全フィールドを更新（`groups` は `JSON.stringify(groupIds)`）

---

#### DELETE `/api/contacts/[id]`

連絡先を削除する。

**処理フロー**
1. `id` と `organizationId` で対象を検索（存在しない → 404）
2. 削除実行（`EmailCampaignRecipient.contactId` は SetNull で保持）

---

### 2.4 招待 API

#### POST `/api/invitations/create`

招待リンクを発行してメール送信する。

**リクエスト**
```json
{ "email": "new@example.com", "role": "VIEWER" }
```

**処理フロー**
1. VIEWER → 403
2. ランダムトークン生成 → ハッシュ化して Invitation 保存（有効期限 72 時間）
3. `GET /api/invitations/accept?token=<rawToken>` の URL を生成
4. メール送信（開発環境ではコンソール出力）
5. `{ ok: true }` を返す

---

#### GET `/api/invitations/accept?token=<token>`

招待を受諾してユーザーを組織に追加する。

**処理フロー**
1. トークンをハッシュ化して Invitation 検索
2. 有効期限切れ・使用済み → エラーリダイレクト
3. 既存 User（email 一致）が存在すれば再利用、なければ新規作成
4. Membership を作成（role は Invitation.role）
5. Invitation.acceptedAt を更新
6. セッション発行 → `/dashboard` にリダイレクト

---

### 2.5 管理者 API

#### PATCH `/api/admin/users/[userId]/password`

**リクエスト**
```json

```

**バリデーション**

- パスワードが 8 文字未満 → 400
- 対象ユーザーが同組織に存在しない → 404

**処理フロー**
1. bcrypt でハッシュ化
2. `User.passwordHash` を更新

## 3. Server Actions 詳細設計

ページ遷移を伴う操作（テンプレートの作成・更新・アーカイブ）は Next.js Server Actions で実装する。

### 3.1 テンプレート作成 (`createTemplateAction`)

```
入力: FormData (name, subject, textBody, htmlBody)
処理:
  1. セッション検証
  2. VIEWER → /dashboard/templates?err=forbidden にリダイレクト
  3. EmailTemplate.create
  4. /dashboard/templates にリダイレクト
```

### 3.2 テンプレート更新 (`updateTemplateAction`)

```
入力: templateId, FormData (name, subject, textBody, htmlBody)
処理:
  1. セッション検証
  2. organizationId + templateId で存在確認
  3. EmailTemplate.update
  4. /dashboard/templates/[id]/edit?ok=1 にリダイレクト
```

### 3.3 テンプレートアーカイブ / 復元

```
archiveTemplateAction: isArchived = true に更新
restoreTemplateAction: isArchived = false に更新
処理後: /dashboard/templates にリダイレクト
```

## 4. メール送信処理 詳細設計

### 4.1 メール送信モード切り替え

`src/lib/email/index.ts` が `EMAIL_MODE` 環境変数を参照し、送信実装を切り替える。

```
EMAIL_MODE=console（デフォルト）
  └─ src/lib/email/sender.console.ts を使用
      → ターミナルにメール内容を出力（実際には送信しない）
      → messageId は "console-<timestamp>" を返す

EMAIL_MODE=smtp
  └─ src/lib/email/sender.smtp.ts を使用
      → Nodemailer で SMTP サーバーに実際に送信する
      → 必要な環境変数: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, MAIL_FROM
```

**コンソール出力フォーマット（`EMAIL_MODE=console`）**

```
=== EMAIL (console) ===
to: <宛先メールアドレス>
subject: <件名>
text: <本文テキスト>
html: <HTML本文（存在する場合）>
=======================
```

**マジックリンクのメール通知**

マジックリンクのメール送信は `EMAIL_MODE` に関わらず、常にコンソール出力する（`src/lib/mail.ts`）。

```
========================================
📩 Magic Link (DEV)
To: <宛先メールアドレス>
URL: <ワンタイムログインURL>
========================================
```

### 4.2 メール配信 Server Action（`sendAction`）

`/dashboard/compose` の Server Action の処理詳細。

```
入力: FormData (ids, templateId, subject, textBody, htmlBody)

1. セッション検証・ロール確認
2. ids をパース（カンマ区切り → 配列、最大 500 件・重複除去）
3. subject, textBody バリデーション（空なら compose 画面にリダイレクト）
4. organizationId スコープで Contact 再取得（URL 改ざん対策）
5. EmailCampaign 作成（status: SENDING）
6. EmailCampaignRecipient を createMany で一括作成
   - email あり → status: PENDING
   - email なし → status: SKIPPED, errorMessage: "email is missing"
7. PENDING の Recipient をループ処理:
   a. 差し込み変数を置換（applyVars 関数）
   b. sendEmail() を呼び出し（EMAIL_MODE に応じてコンソールまたは SMTP）
   c. 成功 → status: SENT, providerMessageId を保存
   d. 失敗 → status: FAILED, errorMessage を保存
8. EmailCampaign を更新:
   - totalCount, sentCount, failedCount, skippedCount を集計
   - failedCount > 0 → status: FAILED, それ以外 → status: SENT
9. /dashboard/compose/result?campaignId=xxx にリダイレクト
```

### 差し込み変数置換関数（applyVars）

```typescript
function applyVars(template: string, contact: Contact): string {
  return template
    .replaceAll("{{name}}", contact.name ?? "")
    .replaceAll("{{companyName}}", contact.companyName ?? "")
    .replaceAll("{{email}}", contact.email ?? "")
    .replaceAll("{{phone}}", contact.phone ?? "");
}
```

## 5. セッション詳細設計

### セッション構造

```typescript
type Session = {
  userId: string;
  organizationId: string;
  role: "ADMIN" | "EDITOR" | "VIEWER";
}
```

### セッションの取得（`getSession`）

1. Cookie から `session` を取得
2. JSON パース
3. 必須フィールド（userId, organizationId, role）が揃っていなければ null を返す

### セッションの発行（`createSession`）

1. Session オブジェクトを JSON 文字列化
2. `HttpOnly: true`, `SameSite: lax`, `Path: /` で Cookie にセット
